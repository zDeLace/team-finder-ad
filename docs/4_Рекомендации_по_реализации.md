# Рекомендации по реализации

Рекомендуется реализовывать по плану ниже, тогда сможете использовать готовые html-шаблоны.
Если хотите изменить детали реализации, указанные в этом уроке, обязательно укажите для ревьюера в `readme.md` какие изменения были произведены и почему.

## Модель данных

> **Все варианты:**
> Названия моделей и полей обязательно должны совпадать с названиями в задании. Иначе html-шаблоны не смогут получить данные из переданных объектов.

### Модель проекта `Project`

* `name`: название проекта. Строка, **обязательное поле**. Максимальная длина - 200 символов.
* `description`: Описание проекта. Текстовое поле, может быть пустым.
* `owner`: Автор проекта. Внешний ключ на модель `User`, **обязательное поле**. У каждого пользователя можно получить список проектов, которыми он владеет при помощи `user.owned_projects`.
* `created_at`: Дата создания проекта. **Обязательное поле**. Поле заполняется текущей датой автоматически при создании объекта.
* `github_url`: Ссылка на github. Поле-ссылка, может быть пустым.
* `status`: Статус проекта. Строка, **обязательное поле**. Может быть только одним из двух вариантов: `[ ("open", "Open"), ("closed", "Closed")]`. Максимальная длина – 6 символов.
* `participants`: Участники проекта. `ManyToMany` связь с моделью `User`, может быть пустым. У каждого пользователя можно получить список проектов, в которых он участвует при помощи `user.participated_projects`.

### Модель пользователя `User`

* `email`: Адрес электронной почты. Поле с типом `email`, **обязательное и уникальное**.
* `name`: Имя пользователя. Строка, **обязательное поле**. Макимальная длина – 124 символа.
* `surname`: Фамилия пользователя. Строка, **обязательное поле**. Макимальная длина – 124 символа.
* `avatar`: Аватарка пользователя. Поле-картинка, **обязательное поле**.
* `phone`: Номер телефона. Строка, **обязательное поле**. Макимальная длина – 12 символа.
* `github_url`: Ссылка на github. Поле-ссылка, может быть пустым.
* `about`: Описание профиля. Текстовое поле, может быть пустым. Максимальная длина – 256 символов.
* `is_active`: Активный пользователь. Булево поле, по умолчанию: `True`.
* `is_staff`: Администратор. Булево поле, по умолчанию `False`.
* Поле, идентифицирующее пользователя – `"email"`
* Может пригодиться реализовать свой `UserManager`, наследующий `django.contrib.auth.models.BaseUserManager`.
* При создании объекта `User` должна генерироваться первоначальная аватарка и записываться в поле `avatar` перед сохранением в БД. Аватарка должна содержать Первую букву имени пользователя на однотонном фоне. Постарайтесь выбрать цвет фона таким образом, чтобы буква была хорошо видна и фон не бросался в глаза излишней контрастностью. Можно выбирать случайный цвет из множества доступных.

Из эндпоинта получите id проекта и измените поле favorites пользователя, приславшего запрос. Если проект добавлен – удалите, а если нет – добавьте. В качестве ответа отправьте такой json объект: `{"status": "ok", "favorited": <текущее состояние проекта для данного пользователя (избранный/не избранный)>}`.

Поэтому в параметрах GET запроса может прийти параметр `filter`. Обрабатывайте его только тогда, когда запрос отправил авторизованный пользователь.

Если `filter` был передан, тогда нужно выбрать из всех пользователей только тех, кто соответствует выбранному фильру, и после этого отрендерить шаблон `/templates/users/participants.html` с таким контекстом: `{"participants": <отфильтрованный queryset пользователей>, "active_filter": <все добавленные в БД навыки>, "active_skill": <выбранный фильтр>}`.

Отличие в том, что все проекты, которые отображаются на этой странице – проекты, добавленные в избранное у данного пользователя, поэтому бэкенд должен обрабатывать запросы только от авторизованных пользователей. Queryset проектов должен содержать проекты, которые пользователь, отправивший запрос, добавил в избранное. В ответ на запрос нужно отрендерить шаблон `/templates/projects/favorite_projects.html` с таким контекстом: `{"projects": <отфильтрованный queryset проектов>}`.

## Страница проекта

### Варианты 1-2

Страница проекта содержит подробную информацию о проекте, id которого указан в эндпоинте GET `/projects/<id проекта>/`. При получении такого запроса бэкенд должен отрисовать шаблон `/templates/projects/project-details.html` с таким контекстом: `{"project": <instance проекта>}`

Если пользователь открыл страницу своего проекта, на UI появляются элементы управления состоянием проекта. А именно, кнопка “Редактировать” (она просто редиректит пользователя на страницу редактирования проекта, о которой будет написано дальше) и кнопка “Завершить проект” (если проект всё ещё открыт).

При нажатии на кнопку “Завершить проект” отправляется запрос POST `/projects/<id проекта>/complete/`. Бэкенд должен проверить, авторизован ли пользователь, является ли он автором проекта и открыт ли проект (статус – “open”). Если все условия выполнены, бэкенд должен отреагировать сменой статуса проекта на “closed” и таким json-ответом: `{"status": "ok", "project_status": "closed"}`.

Если пользователь открыл страницу чужого проекта, то вместо элементов управления состоянием проекта он увидит кнопку “Участвовать” или “Отказаться от участия”. Эти кнопки заменяют друг друга в зависимости от того, является ли пользователь участником этого проекта или нет. Но по части взаимодействия с бэкендом они работают одинаково – отправляют POST запрос `/projects/<id проекта>/toggle-participate`. Бэкенд должен обрабатывать только запросы от авторизованных пользователей. Получите проект из эндпоинта, если в `participants` проекта есть пользователь, отправивший запрос – удалите, а если нет – добавьте.

### Вариант 3

На странице также есть раздел “Необходимые навыки”, это особенность Вашего варианта. Пользователь, открывший страницу чужого проекта, может просто просматривать навыки, а автор проекта – управлять ими.

#### Автодополнение

Автор проекта может нажать на кнопку “Добавить навык”. После этого появится поле ввода. Если начать вводить название навыка, UI будет предлагать выбрать навык из уже добавленных в систему. Это реализовано при помощи запроса GET `/projects/skills/?q=<начало названия навыка>` к бэкенду. В ответ на такой запрос бэкенд должен ответить списком из 10 первых (в алфовитном порядке) навыков, которые добавлены в базу данных и начинаются с подстроки `q` из параметров запроса. Формат возвращаемых данных:

```json
[
  {"id": <id навыка>, "name": <название навыка>},
  { … },
  …
]
```

Позаботьтесь об эффективности: используйте возможности Django ORM. Работайте не с Python списками, а с queryset. В таком случае в оперативную память будут загружены только необходимые данные, а ненужные будут отфильтрованы на уровне SQL.

#### Добавление навыка

Если пользователь хочет добавить проекту навык, которые ранее не был добавлен в базу, ему будет предложено “Создать <навык>”. Если пользователь выберет этот вариант, к бэкенду отправится запрос POST `/projects/<id проекта>/skills/add`. В теле запроса может быть либо параметр `skill_id`, либо параметр `name`.

Для начала проверьте, что пользователь авторизован и имеет право выполнять это действие (является автором проекта).

Если был передан параметр `skill_id`, значит пользователь собирается добавить к своему проекту один из навыков, которые уже присутствуют в базе данных. Установите связь между проектом и этим навыком. Id проекта получайте из эндпоинта.

Если был передан параметр `name`, значит пользователь собирается добавить навык, который ещё не записан в базу данных. Создайте его и установите связь с проектом.

В качестве ответа на запрос нужно вернуть такой json-объект:
`{"skill_id": <id навыка, найденного в БД или созданного>, "created": <булево значение, показывает, был ли создан новый навык в БД>, "added": <булево значение, показывает, была ли установлена связь навыка с проектом>}`.

Связь между навыком и проектом может быть не установлена, например, в том случае, когда пользователь пытается добавить к своему проекту навык, который уже есть в необходимых навыках этого проекта.

#### Удаление навыка

Добавленные в проект навыки можно удалить, нажав на крестик. При этом отправляется запрос POST `/projects/<id проекта>/skills/<id навыка>/remove/`. Бэкенд при получении такого запроса должен проверять, есть ли у пользователя права на удаление навыка (автор проекта), существует ли переданный проект, существует ли переданный навык, добавлен ли этот навык в этот проект. Если все проверки прошли успешно – удаляет навык из атрибута `skills` проекта. Причём сам навык остаётся добавленным в БД приложения и будет доступен для добавления в другие проекты.

## Форма регистрации

Все оставшиеся страницы, включая эту, должны работать при помощи Django-форм (`from django.forms import Form` или дочерние формы, например `ModelForm`, и т.д.). Принцип работы таких страниц-форм следующий:

*   Если пользователь отправляет GET запрос к бэкенду, значит он только открывает страницу через свой браузер. В таком случае нужно отрисовать HTML шаблон, передав в контекст нужную форму. Формат формы будет подробно расписан ниже для каждой страницы.
*   Если пользователь отправляет POST запрос к бэкенду, значит он уже заполнил форму и хочет отправить результаты. В таком случае бэкенд должен провалидировать форму и, если валидация успешна, произвести необходимые манипуляции с данными в БД.

Чтобы зарегистрировать аккаунт, нужно обратиться по адресу `/users/register/`. В ответ на GET запрос бэкенд должен отрендерить шаблон `/templates/users/register.html` передав в контекст `{"form": <форма регистрации>}`. Форма регистрации должна содержать поля `name`, `surname`, `email` и `password`, как в модели пользователя.

В ответ на POST запрос нужно создать нового пользователя в соответвтии с полученными данными, авторизировать текущего пользователя и переадресовать его на главную страницу (`/projects/list`). Если были переданы невалидные данные, снова отрисовать форму.

## Форма авторизации

Чтобы войти в аккаунт, нужно обратиться по адресу `/users/login/`. В ответ на GET запрос бэкенд должен отрендерить шаблон `/templates/users/login.html` передав в контекст `{"form": <форма авторизации>}`. Форма авторизации должна содержать поля `email` и `password`.

В ответ на POST запрос нужно произвести попытку авторизации пользователя в соответвтии с полученными данными. Если успешно, то переадресовать его на главную страницу (`/projects/list`). Иначе добавить к форме ошибку формы “Неверный email или пароль” и снова отрисовать эту форму.

## Форма редактирования профиля

Форма редактирования профиля содержит поля `name`, `surname`, `avatar`, `about`, `phone`, `github_url`, как в модели пользователя.

### Валидация телефонного номера

*   Должен сохраняться только номер телефона в одном из двух форматов: либо `8XXXXXXXXXX`, либо `+7XXXXXXXXXX`. `Х` - цифры.
*   Номера телефона должны быть уникальны, у разных пользователей не могут повторяться номера телефона. В том числе если у одного пользователя `8...`, а у другого `+7...`
*   Должна высвечиваться ошибка формы, если номер телефона неправильного формата или совпадает с другими пользователями.
*   Можно при сохранении номера телефона приводить номера к одному формату, например `8...` -> `+7...`

### Валидация поля Ссылка на Github

*   Ссылка на github должна быть валидной ссылкой.
*   Ссылка должна вести именно на github.

При получении GET запроса, верните отрисованный шаблон `/templates/users/edit_profile.html`, передав в контекст форму. `{"from": <форма>}`. Поля формы заполните текущими данными о пользователе.

При получении POST запроса провалидируйте данные и сохраните изменения. Если всё прошло успешно, переадресуйте пользователя на `/users/<id пользователя>`, иначе верните форму с заполненными ошибками.

## Форма смены пароля

Форма изменения пароля содержит поля `old_password`, `new_password1`, `new_password2`.

При получении GET запроса, верните отрисованный шаблон `/templates/users/change_password.html`, передав в контекст форму. `{"from": <форма>}`.

При получении POST запроса провалидируйте данные и сохраните изменения. Если всё прошло успешно, переадресуйте пользователя на `/users/<id пользователя>`, иначе верните форму с заполненными ошибками.

## Выход из аккаунта

У авторизованных пользователей в боковом меню есть кнопка, позволяющая выйти из аккаунта. При нажатии на эту кнопку отправляется GET запрос на `/users/logout/`. При получении этого запроса разлогиньте пользователя и переадресуйте на `/projects/list/`, главную страницу.

## Форма создания и редактирования проекта

Процессы создания и редактирования проекта практически не отличаются, поэтому за создание и редактирования проекта отвечает один и тот же HTML-шаблон: `/templates/projects/create-project.html`. В контекст этого шаблона нужно будет передать не только форму, но ещё и bool флаг `is_edit`:
`{"form": <форма создания/редактирования>, "is_edit": <флаг>}`

С UI на бэкенд будут приходить разные запросы для создания и редактирования: `/projects/create-project/` и `/projects/<id проекта>/edit/`. Вы можете написать общую view для обработки этих процессов или две разные.

Параметры Django-формы тоже общие (можно использовать одну форму):
`name`, `description`, `github_url`, `status`

Поле `status` должно быть выпадающим списком с двумя вариантами: “Открыт” или “Закрыт”.

### Валидация поля Ссылка на Github

*   Ссылка на github должна быть валидной ссылкой.
*   Ссылка должна вести именно на github.

### Создание проекта

Форма должна быть доступна по адресу `/projects/create-project/`.

При получении GET запроса, верните отрисованный шаблон `/templates/projects/create-project.html`, передав в контекст форму и `is_edit=False`.

При получении POST запроса провалидируйте данные и сохраните в БД новый проект. При сохранении проекта текущий пользователь проставляется в автора проекта и в участники проекта. Если всё прошло успешно, переадресуйте пользователя на страницу только что созданного проекта `/projects/<id проекта>`, иначе верните форму с заполненными ошибками.

### Редактирование проекта

Форма должна быть доступна по адресу `/projects/<id проекта>/edit/`.

При получении GET запроса, верните отрисованный шаблон `/templates/projects/create-project.html`, передав в контекст форму и `is_edit=True`. Поля формы заполните текущими значениями полей проекта.

При получении POST запроса провалидируйте данные и сохраните в БД отредактированный проект. Если всё прошло успешно, переадресуйте пользователя на страницу только что созданного проекта `/projects/<id проекта>`, иначе верните форму с заполненными ошибками.
